### Codex/GPT 工作流启发

1. Build Week 进入评审期，提交物要从“能演示”升级到“可复跑”

OpenAI Devpost 的日程显示，2026-07-21 17:00 PDT 截止提交，2026-07-22 09:00 PDT 开始评审。要求不只是 demo，还包括工作项目、项目说明、公开视频、代码仓库、README、运行说明，以及说明 Codex 和 GPT-5.6 在哪里加速了工作。AI Radar 应吸收这个标准: 每日发布也要能被复查，而不是只留一段观点。

2. Codex Micro 的核心不是键盘，而是 agent 状态可见化

Codex Micro 的信号不是“多了一个小硬件”，而是多条 agent thread 需要被看见、切换、接受、拒绝、暂停和调节。如果 agent 状态只藏在长会话里，用户很容易漏掉等待确认、错误和危险动作。软件侧也可以复用同一逻辑: 每个 Codex task 都要有状态、下一步、风险级别和验收物。

3. 浏览器 agent 能力要和测试证据区分

Copilot browser tools 进入 GA 后，agent 可以打开网页、点击、填写、读 console、截图和反馈结果。这对前端调试很有用，但它仍不是完整测试套件。AI Radar 网站发布前可以用浏览器工具辅助检查，但验收仍要保留静态检查、JSON parse、字段同步、Vercel Ready 和公开数据边界。

4. 手机派工正在成为主入口之一

GitHub Mobile 让用户从手机端处理 PR 评论并派给 Copilot cloud agent，OpenAI 的 Codex Remote 也强调手机选择 host、workspace、branch 和 worktree。方向一致: 手机适合捕捉、创建、补上下文、看状态、批准低风险动作；Mac 仍负责 diff、测试、commit、push、deploy。

5. 预算和 session 限额进入 agent 工作流

GitHub 的 AI credit pools 和 session limits 说明，agent 不是无限劳动力，而是会消耗预算的执行系统。个人自动化也应记失败重跑、卡点、source-card 数、knowledge 数和发布状态；未来可以补一个“高成本失败原因”字段。

6. repo overview 是 onboarding 入口，不是架构真相

Copilot repository overview 把“第一次打开仓库时先问它是什么”产品化。它适合做入门地图，但不能替代 README、测试、目录阅读和维护者判断。AI Radar 可复用点是: 每次接手项目时，先生成概览，再用代码和验证命令校验概览。

### iPhone + Mac + Codex 协作玩法

今日推荐链路:

1. iPhone 用 ChatGPT / Shortcuts / mobile Codex 捕捉链接、语音想法、PR 评论和待办。
2. iPhone 只派发低风险动作: 生成草稿、补充上下文、启动只读分析、请求修复建议。
3. Codex 在 Mac 工作区执行: 写 Markdown、JSONL、站点数据、跑 node check、生成 diff。
4. Mac 做终审: 检查公开字段、source-card 数、report-data 同步、GitHub commit、Vercel Ready。
5. 控制台快捷键只绑定只读或可逆动作；accept changes、commit、push、deploy 必须保留二次确认。

### 可沉淀知识卡片

#### 卡片 1: Agent 控制台先管状态，再管快捷键

- 概念: 多条 agent thread 并行后，最先需要的是可见状态、等待输入、错误、完成和风险级别，而不是更多快捷键。
- 适用场景: Codex Micro、Codex Desktop、AI Radar、Claude Code、Copilot cloud agent。
- 操作步骤: 定义状态枚举 -> 标注风险级别 -> 只给低风险动作快捷入口 -> 高风险动作保留确认 -> 每次发布后回填状态。
- 风险: 只做快捷键不做状态，会把误触和漏审放大。
- 原文: https://www.theverge.com/ai-artificial-intelligence/965901/openai-hardware-codex-micro-launch

#### 卡片 2: Hackathon 提交标准可迁移成自动化验收标准

- 概念: OpenAI Build Week 要求 working project、demo、repo、README、运行说明和 Codex 使用说明，这也是个人自动化交付的最低可复查标准。
- 适用场景: AI Radar、原型、自动化脚本、可公开网页。
- 操作步骤: 写清目标 -> 留代码仓库 -> 留运行说明 -> 留验证命令 -> 留 agent 决策点 -> 发布后记录 Ready 证据。
- 风险: 只有截图或口头总结，无法复跑也无法复盘。
- 原文: https://openai.devpost.com/?ref_medium=discover

#### 卡片 3: 移动端适合派工，不适合终审

- 概念: iPhone 可以创建任务、补上下文、处理低风险评论；最终 diff、测试、提交和部署仍应在 Mac 验收。
- 适用场景: Codex Remote、GitHub Mobile Copilot、iPhone Shortcuts。
- 操作步骤: 手机捕捉 -> 手机派工 -> Codex 执行 -> Mac 看 diff -> Mac 跑验证 -> Mac 发布。
- 风险: 在手机上做 merge、push、deploy 容易漏看边界和副作用。
- 原文: https://github.blog/changelog/2026-07-21-github-mobile-fix-pull-request-comments-with-copilot-cloud-agent/

#### 卡片 4: Agent 成本要加 session 护栏

- 概念: agent 工作流需要预算、session 限额和失败原因记录，避免自动化长时间消耗资源却没有可复用产物。
- 适用场景: Copilot CLI/SDK、Codex 长任务、批量采集、自动修复。
- 操作步骤: 设定任务上限 -> 记录重试次数 -> 记录失败原因 -> 只把成功且可复用结果入 knowledge。
- 风险: 没有限额会奖励“跑得久”，而不是奖励“产物有效”。
- 原文: https://github.blog/changelog/2026-07-01-set-ai-credit-session-limits-in-copilot-cli-and-sdk/

## 白话版

今天的 AI 变化像办公室从“一个实习生坐在你旁边”升级成“你有一排会干活的人坐在工位上”。问题不再只是他们会不会写代码，而是你怎么知道谁在忙、谁卡住、谁等你拍板、谁已经做完、谁可能要花太多钱。

OpenAI Build Week 今天进入评审，就像比赛从“大家都在赶作品”进入“评委开始查作业”。评委不只看你嘴上说做了什么，还要看项目能不能跑、有没有 repo、有没有 demo、README 写没写清楚、Codex 和 GPT-5.6 到底帮了哪里。AI Radar 也应该学这套: 每天不是写一篇漂亮总结就完事，而是要有文件、数据、验证命令、commit 和部署证据。

Codex Micro 像一个放在桌上的“任务总控台”。按钮不是重点，重点是它告诉你: 这条 Codex 任务在跑、那条等你确认、另一条出错了。真正有用的不是把所有危险动作都绑成一键执行，而是先把状态摆出来，再决定哪些按钮能按。

GitHub Copilot 让手机上修 PR 评论、在 VS Code 里操作浏览器、给 AI credit 设上限，这些都在讲同一件事: AI 同事越来越能动手，所以你更要管门禁、预算和验收。手机像前台，可以收件、登记、派工；Mac 像财务和法务，最后盖章前要看清楚 diff、测试和发布结果。

所以今天最重要的做法不是买不买 Codex Micro，而是给自己的 AI 工作流画一张控制台: 哪些入口能收任务，哪些动作能自动做，哪些必须你确认，哪些结果才能进长期知识库。没有这张图，AI 越勤快，你越容易漏审。

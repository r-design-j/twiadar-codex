### Codex/GPT 工作流启发

1. Presence 的关键不是“客服 agent”，而是生产岗位说明书

OpenAI Presence 明确说，每个部署从一个具体岗位开始，例如账单、理赔、IT 服务请求。agent 只拿这个岗位需要的知识和系统权限，公司定义政策、批准动作和人工接管点。这个结构可以直接迁移到个人自动化: 每个 Codex task 也需要岗位卡，而不是一句“帮我处理一下”。

2. 评测要在上线前、上线中、上线后持续存在

Presence 把模拟、grader、guardrails、production sessions、escalations 和 quality signals 放在同一闭环里。个人工作流也应如此: 开始前明确验收命令，执行中记录异常，发布后记录 Ready 和失败原因。今天的 AI Radar 继续保留 node check、JSON parse、字段同步、GitHub push、Vercel inspect。

3. Codex 的角色正在变成“改进生产 agent 的工程师”

Presence 里 Codex 会调查生产会话和升级记录，提出可测试、可批准的改进。这个信号很重要: Codex 不只是写代码，也会变成“看运行数据、提出流程修补、生成候选变更”的维护者。个人知识库也应保留 candidate -> test -> approve -> active 的闸门。

4. GitHub impact dashboard 把 agent 成熟度产品化

GitHub 新仪表盘按 adoption phase 划分 Code-first、Agent-first、Multi-agent/Copilot app 和 Passive，不再只看活跃用户。个人使用 AI 也可以从“用了多少次”转向“哪类任务进入多 agent、哪类任务还停留在聊天、哪些产物被复用”。

5. 模型菜单正在变成调度问题

Gemini 3.6 Flash 进入 Copilot，GPT-5.6 有 Sol / Terra / Luna 和 max / ultra，Copilot 还有按任务自动选模型。这说明下一步重点不是手工崇拜某个模型，而是给任务选择能力、成本、延迟和风险档位。高风险发布仍需人工终审，低风险采集和格式转换可用便宜模型。

6. 媒体/知识型组织的 AI 用法值得迁移

新闻机构案例里有 overnight scan、public filings 搜索、style GPT、public meeting transcript scoring、Slack Data Scouts、MCP 连接数据湖。这些不是“写一篇文章”的玩法，而是线索发现、证据结构化、风格一致性、业务决策入口。AI Radar 可以继续向“公共线索评分系统”演进。

### iPhone + Mac + Codex 协作玩法

今日推荐链路:

1. iPhone 捕捉链接、X 帖、语音想法和待办，只进入候选池。
2. Codex 在 Mac 上把候选池拆成 source-card: 来源、主题、为什么重要、可执行 takeaway、confidence。
3. 对每条高信号任务补岗位卡: 目标、允许资料、允许动作、审批点、验收命令。
4. Mac 执行文件修改、JSON 校验、diff、commit、push、Vercel inspect。
5. knowledge 只收已经能复用的操作链路；新闻摘要留在 daily，不直接进入长期记忆。

### 可沉淀知识卡片

#### 卡片 1: Agent 上生产前先写岗位卡

- 概念: 每个 agent 先绑定一个具体岗位，而不是泛化地“什么都能做”。
- 适用场景: OpenAI Presence、AI Radar、客服 agent、个人自动化。
- 操作步骤: 写任务目标 -> 列可读资料 -> 列允许动作 -> 列需要批准的动作 -> 列人工接管条件 -> 列验收命令。
- 风险: 没有岗位卡，agent 会在权限、目标和验收上漂移。
- 原文: https://openai.com/index/introducing-openai-presence/

#### 卡片 2: Agent 成熟度要按阶段衡量

- 概念: 从 Chat、Code-first、Agent-first 到 Multi-agent，每个阶段的指标不同。
- 适用场景: GitHub Copilot、Codex、AI Radar 月报、个人工作流复盘。
- 操作步骤: 给任务标注 entry / execution / validation / reuse -> 统计可复用产物 -> 记录失败原因 -> 月底清理过时卡片。
- 风险: 只看使用次数会奖励噪声，不会奖励真正节省时间的链路。
- 原文: https://github.blog/changelog/2026-07-22-new-copilot-usage-metrics-impact-dashboard/

#### 卡片 3: Programmatic Tool Calling 适合过滤中间噪声

- 概念: 重复、工具密集、数据量大的流程，不必把每个中间结果都交给模型阅读。
- 适用场景: Responses API、采集筛选、日志分析、网页抽取、批量验收。
- 操作步骤: 把工具白名单化 -> 让程序先过滤中间数据 -> 只把关键证据回传模型 -> 比较成功率、token、延迟和成本。
- 风险: 只优化 token 但降低最终质量，是假优化。
- 原文: https://developers.openai.com/api/docs/guides/latest-model

#### 卡片 4: 新闻工作流可迁移为线索评分系统

- 概念: 新闻机构把 AI 用在线索发现、公开资料结构化、风格校验、知识库查询和 Slack 决策入口。
- 适用场景: AI Radar、行业监控、知识库、公开资料研究。
- 操作步骤: 扫描公开来源 -> 结构化证据 -> 打分排序 -> 人工编辑判断 -> 沉淀可执行卡片。
- 风险: 如果只总结文章，不评分和沉淀动作，就无法形成复利。
- 原文: https://openai.com/index/how-news-organizations-are-using-ai/

## 白话版

今天这波变化像公司终于不再说“我们请了一个很聪明的新同事”，而是开始给这个同事发工牌、写岗位职责、设审批流、做绩效考核。

OpenAI Presence 像是把 AI 员工放进真正的客服、理赔、IT 支持岗位。它不是让 AI 随便发挥，而是规定它能看哪些资料、能点哪些系统按钮、什么时候必须叫真人、上线前怎么模拟考试、上线后怎么根据投诉和失败记录改进。这里 Codex 的角色像内部流程工程师: 看哪里卡住，然后提改法，但改法还要测试和批准。

GitHub 的 Copilot impact dashboard 像老板不再只看“今天有多少人打开了 AI”，而是看大家到了哪一档: 有的人只是用 AI 补代码，有的人已经让 agent 修 PR，有的人开始多 agent 并行。对你自己的工作流也一样，真正要看的不是“我今天用了几次 Codex”，而是“哪些任务已经能派出去、哪些还能自动验收、哪些结果后来真的复用了”。

GPT-5.6、Programmatic Tool Calling 和模型菜单变化，像厨房不再只比谁刀快，而是在设计后厨流水线。切菜、洗菜、摆盘、上菜，不一定都要大厨亲自看一遍。简单中间步骤可以交给程序过滤，最后关键判断再交给强模型和你本人。

iPhone 和 Mac 的分工也更清楚了: iPhone 像前台和派单员，负责把线索收进来、补上下文、派低风险任务；Mac 像主管，负责看最终文件、测试、commit、push 和部署。今天最该补的不是更多工具，而是一张“谁能做什么、什么时候要你盖章”的超市上架清单。

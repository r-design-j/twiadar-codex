### Codex/GPT 工作流启发

1. ChatGPT Work 是“工作台”，Codex 是“执行工位”

OpenAI 最新叙事不再只是某个模型更强，而是把多个能力收成一个工作台: Projects 放上下文，Connectors 接外部资料，Record 负责会议和语音输入，Deep Research 负责长调查，Codex 负责代码和文件执行。对个人工作流来说，重点不是“全部打开”，而是给每类任务一个入口: iPhone 捕捉线索，ChatGPT Work 管上下文，Codex 做可验证产出，Mac 做最终验收。

2. Web Search 要从“搜到结果”升级为“来源可审计”

Responses API 的 Web Search guide 强调新工具、筛选、来源控制和结构化输出。这对 AI Radar 很关键: 每条 source-card 不能只写“看到一个说法”，而要保留公开链接、作者、来源类型、为什么可信、能不能转成动作。后续如果做半自动采集，第一阶段只做候选池，不直接发布。

3. Codex 手机端不是小号 IDE，而是派工入口

手机端最适合做三件事: 捕捉公开链接、创建或补充 Codex task、轻量审批。复杂 diff、部署、公开发布和跨文件复核仍回到 Mac。这个分工能降低手机误操作，也避免把长任务塞进小屏幕。

4. coding-agent 价值需要指标化

arXiv 的 CLI coding agent 采用研究提醒: “感觉省时间”不够。AI Radar 继续把 sourceCards、topSignals、knowledgeCards、验证命令、部署状态写入站点数据，后续还可以补充返工率、失败原因和用户是否实际采用。

5. Agent 边界要写在任务前，不要事后补救

Claude Code hooks 文档和 agentic misalignment 论文共同指向一个结论: 自动化越强，越要把可做、不可做、必须确认的点前置。AI Radar 当前保持只读采集；写文件、commit、push、deploy 是发布阶段；关注、点赞、转发、评论、私信、授权应用继续禁止。

### iPhone + Mac + Codex 协作玩法

今日推荐链路:

1. iPhone 捕捉: 看到 OpenAI、X、GitHub、论文或视频的公开链接，只复制链接或创建 Codex task。
2. ChatGPT Work 收纳: 把主题放进 Project，关联公开资料和本地可公开 Markdown，不把私信、cookie、截图、非公开账号信息放进公开站点。
3. Codex 执行: 只读采集 -> 生成 source-cards -> 写日报 -> 更新站点数据 -> 运行验证命令。
4. Mac 验收: 看日报、Top 5 原文、source-card 条数、JSON 校验、Vercel Ready，再决定是否继续。
5. 知识沉淀: 只把“能复用的操作链路”写进 knowledge，不把一次性新闻堆进去。

### 可沉淀知识卡片

#### 卡片 1: AI 工作台要分四个区

- 概念: ChatGPT Work 类能力适合承载收集、执行、验收、沉淀四个区。
- 适用场景: 日报、研究、Codex 执行、知识库。
- 操作步骤: iPhone 捕捉链接 -> ChatGPT Project 收上下文 -> Codex 产出文件 -> Mac 验收并发布 -> knowledge 沉淀可复用卡片。
- 风险: 不要把非公开资料写入公开站点。
- 原文: https://openai.com/index/chatgpt-for-your-most-ambitious-work/

#### 卡片 2: Web Search 结果要可追溯

- 概念: 搜索不是结论，公开来源链接和筛选条件才是可复核证据。
- 适用场景: AI Radar、竞品观察、公开资料研究。
- 操作步骤: 记录 query -> 保留原文 URL -> 标注 source 类型 -> 写 why_it_matters -> 写 actionable_takeaway -> 丢弃无证据转载。
- 风险: 不要把搜索摘要当原文。
- 原文: https://developers.openai.com/api/docs/guides/tools-web-search

#### 卡片 3: 手机派工，Mac 终审

- 概念: iPhone 负责启动和纠偏，Mac 负责看证据和不可逆动作。
- 适用场景: Codex task、日报发布、PR review、公开网站更新。
- 操作步骤: 手机创建任务 -> Codex 执行 -> Mac 看 diff、验证命令和部署状态 -> 决定发布。
- 风险: 不要在手机上做复杂 diff 或公开发布终审。
- 原文: https://learn.chatgpt.com/docs/changelog

#### 卡片 4: Agent 采用要有本地指标

- 概念: Agent 有没有省时间，不能只靠感觉，要看输出、验证、返工和采用。
- 适用场景: 自动化日报、coding agent、知识库维护。
- 操作步骤: 记录 sourceCards/topSignals/knowledgeCards -> 记录验证命令 -> 记录发布状态 -> 记录失败原因 -> 月底复盘哪些动作真的继续使用。
- 风险: PR 数或任务数只是代理指标，不等于真实价值。
- 原文: https://arxiv.org/abs/2607.01418

## 白话版

今天的重点可以理解成: 以前你像是在跟一个很聪明的人聊天，现在更像是在搭一个小型办公室。

ChatGPT Work 像办公室前台和资料柜，帮你把项目、资料、会议记录和研究放在一起。Codex 像坐在电脑前真正干活的人，负责改文件、整理日报、跑验证。iPhone 像遥控器，适合发任务、补一句话、看状态。Mac 像验收台，适合看清楚文件、链接、验证结果和发布状态。

Web Search 这件事也要升级。以前是“我搜到了一个说法”，现在要变成“我从哪里搜到、原文是谁、链接在哪、为什么可信、我能拿它做什么”。这就像买东西不能只听导购说好，要看商品页、参数、评价和退换规则。

coding agent 的价值也不能只靠感觉。不能只说“它好像帮我省时间了”，要看它每天产出了几张 source-card、几条 Top signal、几张知识卡、有没有跑验证、有没有部署成功、有没有返工。

最后，越能干的 agent 越要先说清楚边界。就像请了一个很厉害的助理，不是让他随便替你发朋友圈、改合同、转账，而是先说清楚: 哪些可以自己做，哪些必须等你点头。

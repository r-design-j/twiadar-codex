2026-07-21 的主线是: AI 工作流正在从“能不能让 agent 干活”进入“怎么把入口、权限、度量、预算、安全和记忆管起来”。OpenAI 侧把 Chat、Work、Codex、Sites、Plugins、移动 Remote 和私有 MCP 逐步收束成跨设备工作台；GitHub 侧把 Copilot coding agent 的 repo 级指标、AI credit pool、code review 环境、security review、agentic autofix 做成治理能力。对个人 AI Radar 来说，今天最可复用的结论不是追新模型，而是把每次 agent 工作拆成四张清单: 入口清单、权限清单、成本/指标清单、知识沉淀清单。

### Codex/GPT 工作流启发

1. 工作台正在替代“单聊天窗口”

OpenAI Release Notes 把 ChatGPT Work、桌面 app、Codex、Sites 和 Plugin Directory 放在同一轮更新语境里。对个人流程的启发是: 不要把所有工作都塞进一个长聊天。收集线索、执行任务、验收结果、发布网站、沉淀知识应该有不同承载物。

2. Agent 采用需要 repo 级或项目级指标

GitHub Copilot usage metrics 进入 repository-level 后，agent 价值不再只靠“感觉省时间”。AI Radar 已经有 sourceCards、topSignals、knowledgeCards；下一步应补“实际复用次数、失败原因、返工点、发布是否 Ready”。这样月底才能判断哪类信号真的值得自动化。

3. Code review agent 的边界越来越工程化

Copilot code review 开始支持从 head branch 读取自定义指令、专用 setup 文件、默认 firewall、独立 runner 配置。这说明 agent review 的质量不只取决于模型，还取决于运行环境、规则入口和网络边界。Codex 任务也应该明确: 读哪个 AGENTS.md，能访问哪些路径，是否联网，跑什么验证。

4. 安全能力从“检查代码”扩展到“让 agent 自己修复，但仍交 PR”

GitHub 的 `/security-review` 和 agentic autofix 都把安全检查推到 agent 工作流里，但共同点是仍然保留 review 和验证。个人自动化不要让 agent 直接“发现问题就改并发布”；更稳的链路是发现 -> 拟修 -> 验证 -> 生成 diff -> Mac 终审。

5. 私有 MCP 的核心不是连接，而是边界

OpenAI Developers 对 Secure MCP Tunnel 的表述重点是保持私有网络边界，同时给 ChatGPT/Codex 一个可检查的请求路径。对本地工作流来说，这直接支持已有安全规则: 不要为了方便把本地服务、知识库或公司内网暴露成公共 endpoint。

6. 长期记忆要有 staging area

Context Engineering cookbook 把记忆拆成结构化 profile、global memory、session memory 和合并评估。AI Radar 的 knowledge 文件也应这么看: source-cards 是候选池，daily 是当天叙事，knowledge 只接收已经能复用的操作链路。

### iPhone + Mac + Codex 协作玩法

今日推荐链路:

1. iPhone 用 ChatGPT mobile / Shortcuts 捕捉链接、语音想法和待办，只做派工与轻量补充。
2. ChatGPT Desktop / Codex 负责执行公开采集、写 Markdown、同步 JSON、跑验证。
3. Plugins / Apps 只接已经明确授权的工具；默认不把 Feishu、邮箱、私有知识库接入日报。
4. Mac 负责终审: 检查 diff、验证命令、GitHub commit、Vercel Ready。
5. knowledge 只写“下次还能复用的做法”，不要把一次性新闻硬塞进长期记忆。

### 可沉淀知识卡片

#### 卡片 1: Agent 工作台要有四张清单

- 概念: agent 能跨设备、跨工具工作后，入口、权限、指标、知识沉淀必须分开管理。
- 适用场景: ChatGPT Work、Codex、Sites、Plugins、AI Radar 自动化。
- 操作步骤: 写入口清单 -> 写权限清单 -> 写指标清单 -> 写 knowledge 准入清单 -> 每次发布后复盘。
- 风险: 没有清单会把“能做”误当“该做”。
- 原文: https://help.openai.com/en/articles/6825453-chatgpt-release-notes

#### 卡片 2: Agent 指标要从组织级下钻到项目级

- 概念: repo 级 Copilot 指标提示，agent 价值要落到具体项目和产出，而不是只看总使用量。
- 适用场景: AI Radar 月报、Codex 自动化、代码库维护。
- 操作步骤: 记录 sourceCards、knowledgeCards、失败原因、返工点、被复用次数、发布状态。
- 风险: 只统计运行次数会奖励噪声，而不是奖励有效复用。
- 原文: https://github.blog/changelog/2026-07-17-repository-level-github-copilot-usage-metrics-generally-available/

#### 卡片 3: 私有 MCP 先守边界，再谈连接

- 概念: MCP 的价值是让 AI 访问工具和数据，但私有服务器不应为了接入而公开暴露。
- 适用场景: 本地桥、知识库、公司内网、私有 API、自动化工具。
- 操作步骤: 默认只读 -> 白名单工具 -> 可检查客户端 -> 不开公网隧道 -> 敏感动作人工确认。
- 风险: 便利入口会变成公网攻击面。
- 原文: https://developers.openai.com/blog/connect-private-mcp-servers-to-openai-products

#### 卡片 4: 长期记忆需要候选池和合并闸门

- 概念: 记忆不是把所有内容追加到文件，而是先暂存候选，再去重、冲突处理和评估。
- 适用场景: AI Radar knowledge、Obsidian second brain、Agents SDK long-term memory。
- 操作步骤: source-card 当候选 -> daily 做解释 -> knowledge 只收高信号卡片 -> 月底清理重复和过时项。
- 风险: 未筛选的记忆会污染 agent 判断。
- 原文: https://developers.openai.com/cookbook/examples/agents_sdk/context_personalization

## 白话版

今天的重点不是“又出了一个更聪明的模型”，而是“AI 同事越来越像一个小团队以后，你要开始当办公室管理员”。

以前你只要问它一句话，它回一句话。现在它能开工、看网页、调工具、改代码、做网站、连插件、用手机遥控。能力变强以后，问题就从“会不会干活”变成“谁能进办公室、能碰哪些柜子、花了多少钱、做完有没有验收、经验有没有入库”。

GitHub 的 repo 级 Copilot 指标就像给每个项目装电表: 不是看全公司用了多少电，而是看哪个项目真的靠 AI 多开了 PR、做了 review、修了安全问题。AI Radar 也应该这样，不只数今天看了多少链接，还要看哪些卡片后来真用上了。

Copilot code review 的 firewall 和自定义 setup 像给审稿同事安排固定工位: 他读哪份规则、能不能上网、能不能跑依赖、用哪台机器，都要写清楚。Codex 也一样，不能只说“你帮我搞定”，要给它边界和验收标准。

私有 MCP 就像公司内部门禁。好工具不代表要把门开到大街上。能用同声传译接进来当然好，但门禁、白名单和审计要在前面，方便不能压过安全。

长期记忆也不是垃圾桶。source-card 像候选简历，daily 像面试记录，knowledge 才是正式员工档案。没筛过的东西都塞进知识库，最后 agent 不是变聪明，而是背了一堆过期便签。

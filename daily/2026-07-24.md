### Codex/GPT 工作流启发

1. 语音是新入口，不是新权限

ChatGPT Voice 已进入桌面端 Work 和 Codex。它适合快速说清目标、补充上下文、让 agent 检查进度，尤其适合 Mac 前台窗口和 appshot 场景。但语音输入天然更容易口语化、含糊和省略条件，因此不应直接绑定 push、deploy、发帖、改权限等不可逆动作。

2. 移动端派工正在产品化

GitHub Mobile 现在可以从失败 Actions check 直接触发 Copilot cloud agent 修复，并生成叠加在原 PR 之上的新 PR。这个模式和 Codex Remote 一致: 手机上做触发和监督，桌面/仓库系统留下 diff、检查和审阅链路。

3. Agent 自动化需要 rationale、confidence、approval

GitHub Issues 的 agent automation controls 把 label、type、assign、close 等变更拆成建议、置信度和审批。对个人自动化也一样: 高置信度的格式整理可以自动做；中低置信度的删除、归档、发布、费用相关动作必须进入待确认队列。

4. Linear 派单说明 agent 正在进入项目管理层

Copilot cloud agent for Linear GA 后，issue 可以直接派给异步 agent，且可选模型、自定义 agent、base/working branch 和 session steering。个人工作流可以借鉴: 每张任务卡都要有目标分支、可执行范围、模型档位和“继续/改方向”的入口。

5. 安全边界要从“命令名”升级到“调用和后果”

Pillar 的 GitPwned 报告指向一个关键点: 允许 `git show` 这个命令名并不等于所有 `git show` 调用都安全。agent 写入的 `.git/config`、工作区配置、脚本、hook、IDE 设置，都可能被宿主工具稍后执行。审批要看参数、写入目标和后续消费者。

6. Shortcuts 适合做结构化 intake

Apple Support 文档说明 iPhone Shortcuts 可用 Apple Intelligence、Private Cloud Compute 或 ChatGPT 扩展模型来处理输入、解析输出、接续后续动作。对 AI Radar，最稳的用法是把链接、语音、提醒、当前页面整理成 source-card 候选，而不是让手机直接发布公开内容。

### iPhone + Mac + Codex 协作玩法

今日推荐链路:

1. iPhone Shortcuts 收集链接、X 帖、语音想法和提醒，用模型整理成候选 source-card。
2. 手机或语音只允许创建任务、补上下文、查看状态、要求 agent 调查失败。
3. Codex 在 Mac 上生成 diff、Markdown、JSONL、站点数据和验证命令结果。
4. 中低置信度动作进入人工确认: 删除、归档、改权限、commit、push、deploy、公开发布。
5. Mac 终审后再 commit、push、Vercel deploy，并把失败原因写回 automation memory。

### 可沉淀知识卡片

#### 卡片 1: 语音入口只做协调，不直接发布

- 概念: Voice 可以启动、检查和转向任务，但不应直接绑定不可逆动作。
- 适用场景: ChatGPT desktop、Work、Codex、Mac appshot、移动远程监督。
- 操作步骤: 语音收集目标 -> Codex 生成计划和 diff -> Mac 验证 -> 人工确认 commit/push/deploy。
- 风险: 口语输入容易省略边界，把“帮我发一下”误解成公开发布。
- 原文: https://help.openai.com/en/articles/6825453-chatgpt-release-notes

#### 卡片 2: Agent 自动化要带 rationale 和 confidence

- 概念: agent 做元数据变更时，应说明理由、置信度和是否等待审批。
- 适用场景: GitHub Issues、AI Radar signals、source-card 去重、候选任务分类。
- 操作步骤: 给每个建议标 action / rationale / confidence / auto_apply_policy -> high 可自动整理 -> medium/low 待用户确认。
- 风险: 没有理由和置信度，自动化会把错误动作做得更快。
- 原文: https://github.blog/changelog/2026-07-23-agent-automation-controls-in-github-issues-in-public-preview/

#### 卡片 3: Sandbox 边界要看“写入后谁会信任”

- 概念: agent 的影响范围包括它能写入的文件，以及宿主系统后来会执行、加载、扫描或信任的内容。
- 适用场景: Codex CLI、Git hooks、`.git/config`、IDE 配置、package scripts、MCP 工具。
- 操作步骤: 列 agent 可写路径 -> 列宿主消费者 -> 审查配置型代码 -> 对危险写入加审批 -> 更新工具白名单为“调用级”。
- 风险: 只按命令名做白名单，会漏掉参数和写入目标造成的副作用。
- 原文: https://www.pillar.security/blog/the-week-of-sandbox-escapes

#### 卡片 4: 手机端 intake 要结构化，不做终审

- 概念: iPhone 适合用 Shortcuts/Voice 把零散输入变成结构化候选，但最终文件、测试和发布仍在 Mac。
- 适用场景: AI Radar、日报线索、PR 修复、提醒转任务、公开链接整理。
- 操作步骤: Shortcuts 采集输入 -> 模型解析字段 -> 写入候选池 -> Codex 执行 -> Mac 终审发布。
- 风险: 手机屏幕和语音都不适合复杂 diff 与公开边界审查。
- 原文: https://support.apple.com/en-gb/guide/iphone/iph78c41eaf8/26/ios/26

## 白话版

今天的变化像你家的“遥控器”突然变多了: 以前主要是 Mac 键盘，现在手机、语音、Linear、GitHub Issues 都能叫 agent 干活。

但遥控器变多，不代表每个遥控器都能开保险柜。语音适合说“帮我看看这个任务卡住哪了”“把这条链接整理成候选卡”；手机适合点一下“让 agent 修一下失败的 CI”；Linear 适合把 issue 派给后台 agent。真正要上架、付款、改门锁的动作，还是得回到 Mac 看清楚 diff、检查结果和发布地址。

GitHub 今天给了一个很实用的方向: agent 做事要带理由、置信度和审批。高置信度的小事可以自动做，比如补标签、整理格式；中低置信度的事情就像超市上架清单里没确认的货，先放待审区，不能直接摆上货架。

Pillar 的安全报告则是在提醒: 不能只看 agent 有没有关在房间里，还要看它能往外递什么纸条。它可能没有直接跑出沙箱，但它写了一个配置文件，等 Git、IDE 或脚本之后一读，就把事办了。所以“安全命令”不能只看名字，要看这次调用写到了哪里、谁会信任它。

今天最该做的不是再加十个入口，而是给入口分级: iPhone 和语音是前台，负责收消息、派单、问进度；Codex 是执行同事；Mac 是主管，负责最后看合同、盖章和发布。

### Codex/GPT 工作流启发

1. 模型选择要产品化

不要再把模型选择写成“用最强模型”。今天开始更实用的写法是: 高难分析、长代码修复、复杂安全复核用 Sol/max 或 ultra；日常日报、一般代码修改、常规知识整理用 Terra；高频摘要、分类、初筛用 Luna。每次自动化都要在 source-card 或任务说明里记录“为什么选这个模型层级”。

2. PTC 是工具流水线，不是判断替代品

Programmatic Tool Calling 适合做可边界化的中间处理: 抓取多个工具结果、去重、过滤、排序、聚合、字段校验。它不适合替代“这条内容值不值得沉淀”“是否可以发布”这种判断。正确链路是: PTC 产出更小、更干净的结构化结果 -> GPT 综合判断 -> 人在 Mac 上复核关键动作。

3. Multi-agent 只在天然并行时使用

Responses API 的 Multi-agent beta 让 subagents 成为一等能力，但这不等于所有任务都要拆。AI Radar 可以拆的只有四类: 采集候选、事实核验、操作链路翻译、发布验证。单个来源精读、单个 bug 修复、单个文档改写仍然用一个 agent 更稳。

4. ChatGPT desktop 是 Mac 验收台

Codex 进入 ChatGPT desktop 后，Mac 的角色更清楚: 看 diff、改 Markdown、读 PR review、跨 repo 查看上下文、跑验证、决定发布。它不是和 iPhone 竞争，而是承接 iPhone 派来的任务，做最终验收。

5. 更强 agent 需要更硬的边界

GPT-5.6 system card 对 agentic coding 任务越过用户意图的风险给出提醒。对个人工作流来说，最实际的动作是把只读采集、写入文件、提交代码、公开发布拆成阶段，并在每个阶段写清 approval、停止条件和验收证据。

### iPhone + Mac + Codex 协作玩法

今日推荐链路:

1. iPhone 捕捉: 看到 OpenAI release、X 长帖、GitHub 项目或论文，只复制公开链接或创建 Codex task。
2. iPhone 控制: 选择 host、workspace、branch/worktree；用 Queue 排队，用 Steer 及时纠偏；只做轻审批。
3. Codex 执行: 用 Sol/Terra/Luna 路由不同难度；用 PTC 清洗结构化中间结果；必要时用 subagents 并行采集和核验。
4. Mac 验收: 在 ChatGPT desktop 里看 Markdown、代码、PR、验证命令、发布状态，再决定是否公开。
5. 知识库沉淀: 只有能变成“点哪里 -> 看到什么 -> 再做什么”的动作进入 knowledge。

### 可沉淀知识卡片

#### 卡片 1: 模型选择是一张工作流白名单

- 概念: 模型不是越强越好，而是按任务路由。
- 适用场景: Codex 长任务、日报自动化、知识库整理、批量摘要。
- 操作步骤: 判断任务难度 -> 选择 Sol/Terra/Luna -> 决定是否需要 max/pro/ultra -> 记录选择理由 -> 跑完后看成本和质量。
- 风险: 把所有任务都交给最贵模型，会让自动化失去成本解释能力。
- 原文: https://openai.com/index/gpt-5-6/

#### 卡片 2: PTC 只接管有边界的工具流水线

- 概念: PTC 是让模型写受限小程序协调工具，不是让模型自动拍板。
- 适用场景: 多来源去重、字段校验、排序、聚合、把大中间结果压缩成小结构。
- 操作步骤: 写清阶段目标 -> 写 eligible tools -> 写输出 schema -> 写 retry/stop condition -> 回交 GPT/人工判断。
- 风险: 不要把审批、发布、不可逆动作放进 PTC。
- 原文: https://developers.openai.com/api/docs/guides/tools-programmatic-tool-calling

#### 卡片 3: ChatGPT desktop 是 Mac 验收台

- 概念: iPhone 控制任务，Mac 验收结果；ChatGPT desktop 把 Codex、Markdown、代码、PR 放在一个入口。
- 适用场景: 日报发布、代码修复、PR review、跨 repo 任务。
- 操作步骤: iPhone 派活 -> Codex 执行 -> Mac 打开 ChatGPT desktop -> 看 diff/文档/验证命令 -> 决定提交或发布。
- 风险: 不要在手机上做复杂 diff 终审。
- 原文: https://learn.chatgpt.com/docs/changelog

#### 卡片 4: 更强 agent 更需要硬边界

- 概念: 模型越会用工具，越要写清可做和不可做。
- 适用场景: Codex 写文件、MCP 工具、私有数据、公开发布、安全相关任务。
- 操作步骤: 拆成只读、写入、提交、发布阶段 -> 每阶段写审批点 -> 跑后检查证据和 diff。
- 风险: 不要让 agent 在模糊授权下跨越用户意图。
- 原文: https://deploymentsafety.openai.com/gpt-5-6

## 白话版

今天的主线可以理解成: 你现在不是只有一个“超级员工”，而是有一个小团队和一张派工表。

Sol 像资深专家，适合最难的问题；Terra 像稳定主力，适合日常工作；Luna 像快手助理，适合批量整理。你不需要让专家去贴标签，也不该让快手助理拍关键决策。

PTC 像流水线上的分拣机。它很适合把一堆来源去重、分类、排序、整理成整齐表格；但“这个东西值不值得进入我的长期知识库”这种判断，还是要交给 GPT 综合判断，再让你在 Mac 上看一眼。

Multi-agent 像临时小组。只有任务真的能拆开，比如一个人找资料、一个人核验、一个人写操作步骤、一个人检查发布，才值得组队。一个人能干好的事，不要硬拉四个人开会。

iPhone 像遥控器。它负责选哪台机器、哪个项目、要不要继续排队、要不要中途纠偏。Mac 像验收台，负责看清楚文件、diff、验证命令和发布状态。

更强的 agent 像更能干的同事。越能干，越要提前说清楚“你可以做到哪一步，哪一步必须等我点头”。这不是不信任它，而是把工作边界变清楚。
